Diagrama do Banco de Dados (PostgreSQL)

TABLE users
---------------------------------------------------------------------------------------------------
| user_id                  | BIGINT PRIMARY KEY     | Chave primária. Auto-incremento.            |
| username                 | VARCHAR(255) UNIQUE    | Nome de usuário único.                      |
| password_hash            | VARCHAR(255)           | Hash da senha (hashing com salt).           |
| salt                     | VARCHAR(255)           | Salt para o hashing da senha.               |
| user_type                | VARCHAR(50)            | Tipo de usuário ('admin', 'usuario').       |
| is_banned                | BOOLEAN                | Indica se o usuário está banido.            |
| failed_login_attempts    | INTEGER                | Contador de tentativas de login falhas.     |
| last_login_attempt_at    | TIMESTAMP              | Data/hora da última tentativa de login.     |
---------------------------------------------------------------------------------------------------

TABLE permissions
---------------------------------------------------------------------------------------------------
| permission_id            | BIGINT PRIMARY KEY     | Chave primária.                             |
| user_id                  | BIGINT NOT NULL        | Chave estrangeira para a tabela 'users'.    |
| resource_id              | BIGINT NOT NULL        | ID do recurso (ex: CRUD).                   |
| read_permission          | BOOLEAN                | Permissão de leitura.                       |
| create_permission        | BOOLEAN                | Permissão de criação.                       |
| update_permission        | BOOLEAN                | Permissão de atualização.                   |
| delete_permission        | BOOLEAN                | Permissão de exclusão.                      |
---------------------------------------------------------------------------------------------------
FOREIGN KEY (user_id) REFERENCES users (user_id)

TABLE user_logs
---------------------------------------------------------------------------------------------------
| log_id                   | BIGINT PRIMARY KEY     | Chave primária.                             |
| user_id                  | BIGINT NOT NULL        | Chave estrangeira para a tabela 'users'.    |
| action                   | TEXT                   | Descrição da ação (ex: 'LOGIN', 'CREATE').  |
| timestamp                | TIMESTAMP              | Data e hora da ação.                        |
| ip_address               | VARCHAR(45)            | Endereço IP de onde a ação foi realizada.   |
---------------------------------------------------------------------------------------------------
FOREIGN KEY (user_id) REFERENCES users (user_id)

TABLE crud_configs
---------------------------------------------------------------------------------------------------
| crud_id                  | BIGINT PRIMARY KEY     | Chave primária.                             |
| name                     | VARCHAR(255) NOT NULL  | Nome do CRUD.                               |
| description              | TEXT                   | Descrição do CRUD.                          |
| linked_table_name        | VARCHAR(255) NOT NULL  | Nome da tabela vinculada no banco de dados. |
| is_public                | BOOLEAN                | Indica se o CRUD está público.              |
| terms_accepted           | BOOLEAN                | Indica se os termos de responsabilidade foram aceitos. |
---------------------------------------------------------------------------------------------------

TABLE report_configs
---------------------------------------------------------------------------------------------------
| report_id                | BIGINT PRIMARY KEY     | Chave primária.                             |
| name                     | VARCHAR(255) NOT NULL  | Nome do relatório.                          |
| description              | TEXT                   | Descrição do relatório.                     |
| source_id                | BIGINT NOT NULL        | Chave estrangeira para a fonte de dados.    |
| report_type              | VARCHAR(50)            | Tipo de relatório ('barra', 'tabela').      |
| config_json              | JSONB                  | JSON com as configurações do relatório.     |
| created_by_user_id       | BIGINT                 | Chave estrangeira para o usuário que criou. |
---------------------------------------------------------------------------------------------------
FOREIGN KEY (created_by_user_id) REFERENCES users (user_id)

Requisitos do Sistema

Requisitos Funcionais (RF)

RF1: CRUD

O sistema deve permitir a criação, leitura, atualização e exclusão de dados.

  • RF1a: O sistema deve permitir a criação de novos registros de dados através de um formulário ou interface específica.
  • RF1b: O sistema deve exibir registros de dados existentes em uma lista ou tabela de forma clara e organizada.
  • RF1c: O sistema deve permitir a modificação de registros de dados existentes, garantindo a integridade dos dados.
  • RF1d: O sistema deve permitir a exclusão de registros de dados, com uma confirmação antes da ação para evitar perdas acidentais.

RF2: Geração e Gerenciamento de Relatórios

O sistema deve permitir a geração e o gerenciamento de relatórios.

  • RF2a: O sistema deve fornecer uma interface para que o usuário selecione os critérios de um relatório (ex: período, tipo de dados).
  • RF2b: O sistema deve processar os dados e gerar um relatório com base nos critérios selecionados.
  • RF2c: O sistema deve permitir que relatórios gerados sejam salvos para acesso futuro.

RF3: Autenticação de Usuários

O sistema deve autenticar usuários com nome de usuário e senha.

  • RF3a: O sistema deve ter uma tela de login onde o usuário insere suas credenciais.
  • RF3b: O sistema deve validar as credenciais do usuário comparando-as com os dados armazenados de forma segura.

RF4: Verificação de Nível de Acesso

O sistema deve verificar o nível de acesso (admin ou usuário) do usuário autenticado.

  • RF4a: Após a autenticação, o sistema deve consultar a permissão associada ao usuário.
  • RF4b: O sistema deve usar o nível de acesso verificado para determinar qual interface deve ser exibida.

RF5: Interfaces Diferenciadas

O sistema deve exibir interfaces diferentes (view_admin e view_usuario) com base no nível de acesso.

  • RF5a: A interface view_usuario deve conter apenas as funcionalidades e informações relevantes para o usuário comum.
  • RF5b: A interface view_admin deve conter todas as funcionalidades e informações de gerenciamento, incluindo as opções de CRUD e relatórios.

RF6: Acesso do Administrador

A view_admin deve permitir a visualização de todas as operações CRUD e relatórios.

  • RF6a: A view_admin deve exibir links ou botões claros para acessar todas as telas de CRUD.
  • RF6b: A view_admin deve ter uma seção dedicada para a geração e visualização de todos os relatórios disponíveis.

RF7: Exportação de Dados para JSON

O sistema deve exportar dados de CRUDs para o formato JSON.

  • RF7a: O sistema deve oferecer um botão ou opção para "Exportar para JSON" nas telas de visualização de CRUD.
  • RF7b: O sistema deve converter os dados selecionados para a estrutura JSON e iniciar o download do arquivo.

RF8: Temporizador de Login

O sistema deve iniciar um temporizador após a tentativa de login e notificar o usuário se a resposta demorar.

  • RF8a: O sistema deve iniciar um contador em segundo plano imediatamente após o envio das credenciais de login.
  • RF8b: Se o tempo exceder um limite pré-definido (ex: 5 segundos), o sistema deve exibir uma mensagem ao usuário, como "O login está demorando mais que o esperado...".

RF9: Limite de Tentativas de Login

O sistema deve limitar o número de tentativas de login e fechar o programa se o limite for atingido.

  • RF9a: O sistema deve manter um contador de tentativas de login falhas por usuário ou endereço IP.
  • RF9b: Se o contador atingir um limite pré-definido (ex: 3 tentativas), o sistema deve bloquear temporariamente o acesso ou fechar o programa.

RF10: Modos de Execução

O sistema deve ter a opção de ser executado nos modos "Privado" ou "Público".

  • RF10a: Durante a inicialização, o sistema deve perguntar ao usuário qual modo de execução ele deseja.
  • RF10b: O modo "Privado" deve restringir o acesso a todas as CRUDs, exigindo autenticação total.
  • RF10c: O modo "Público" deve permitir acesso limitado a algumas funcionalidades sem autenticação.

RF11: Ativação de CRUDs Públicas com Consentimento Legal

O sistema deve exigir a ativação manual de cada CRUD pública através de um botão, com um pop-up de aviso legal e um temporizador de espera.

  • RF11a: O sistema deve apresentar um botão ou toggle "Tornar Público" para cada CRUD ou conjunto de dados que pode ser publicado.
  • RF11b: Ao clicar no botão, o sistema deve exibir um pop-up com um resumo claro das responsabilidades legais e um termo de consentimento que o usuário deve aceitar.
  • RF11c: O pop-up deve iniciar um temporizador de 15 segundos. A opção de aceitar e continuar com a postagem só deve ser habilitada após a espera de 15 segundos, para garantir que o usuário leia a mensagem.
  • RF11d: Após o temporizador, o usuário deve clicar em "Aceitar e Publicar" para confirmar a ação. A postagem só será tornada pública após essa confirmação.
  • RF11e: O sistema deve registrar no log de atividades (RF15) que o usuário consentiu com os termos de responsabilidade legal e publicou a CRUD, incluindo a data e o horário da ação.

RF12: Instruções de Importação/Exportação

O sistema deve fornecer instruções claras para a exportação e importação de dados.

  • RF12a: O sistema deve ter uma seção de ajuda ou um tooltip que explique o processo de importação e exportação de dados.
  • RF12b: As instruções devem detalhar o formato e a estrutura esperada do arquivo de dados (ex: JSON).

RF13: Aceitação de Termos

O sistema deve aceitar as respostas do usuário para as perguntas de responsabilidade legal e CRUDs públicas.

  • RF13a: O sistema deve apresentar termos e condições ou perguntas de responsabilidade legal ao usuário, provavelmente no primeiro acesso.
  • RF13b: O sistema deve registrar a aceitação do usuário para futuras verificações.

RF14: Alteração e Redefinição de Senha

O sistema deve permitir que usuários alterem suas próprias senhas e deve incluir um fluxo de "esqueci a senha" para redefinição.

  • RF14a: A view_usuario e view_admin devem ter uma opção para "Alterar Senha".
  • RF14b: A tela de login deve ter uma opção "Esqueci a Senha" que inicia o processo de redefinição.
  • RF14c: O sistema deve validar a nova senha com base em regras de segurança (ex: tamanho mínimo, caracteres especiais).

RF15: Histórico de Ações

O sistema deve registrar um log de atividades do usuário (login, logout, CRUDs), e o administrador deve poder visualizar esse log.

  • RF15a: O sistema deve registrar a data, hora, usuário e a ação realizada em um banco de dados ou arquivo de log.
  • RF15b: A view_admin deve ter uma tela para exibir o log de atividades de forma filtrável e organizada.

RF16: Busca Avançada

O sistema deve fornecer uma funcionalidade de busca avançada que permite filtrar dados por múltiplos critérios.

  • RF16a: O sistema deve ter um campo de busca que aceite múltiplos critérios (ex: nome: "Fulano" ou data > "2023-01-01").
  • RF16b: A busca deve ser rápida e eficiente, mesmo com grandes volumes de dados.

RF17: Geração de Relatórios em Múltiplos Formatos

O sistema deve permitir a exportação de relatórios para os formatos PDF e CSV.

  • RF17a: A tela de relatórios deve ter opções claras para exportar para PDF e CSV.
  • RF17b: A exportação para PDF deve formatar o relatório para impressão, enquanto a exportação para CSV deve criar um arquivo de texto com valores separados por vírgula.

RF18: Permissões Granulares

O sistema deve permitir que o administrador atribua permissões específicas a usuários, como permissão de "leitura" em uma CRUD específica.

  • RF18a: A view_admin deve ter uma tela de "Gerenciamento de Usuários" para editar as permissões.
  • RF18b: O administrador deve poder selecionar se um usuário tem permissão para Criar, Ler, Atualizar ou Excluir para cada tipo de dado (CRUD).

RF19: Gerenciamento de Usuários Públicos

A view_admin deve ter uma funcionalidade para banir usuários do modo "Público".

  • RF19a: O sistema deve armazenar uma lista de usuários banidos no modo "Público" para negar futuras tentativas de login ou acesso.
  • RF19b: O admin deve poder adicionar e remover usuários da lista de banidos.

RF20: Sistema de Denúncias

O sistema deve permitir que usuários reportem atividades de outros usuários considerados "malfeitores".

  • RF20a: O sistema deve ter um mecanismo para coletar informações sobre a denúncia (motivo, usuário denunciado, etc.).
  • RF20b: A view_admin deve incluir um painel para que o administrador possa revisar e agir sobre as denúncias recebidas.
  • RF20c: O sistema deve notificar o usuário que fez a denúncia sobre o status da sua solicitação.

Requisitos Não Funcionais (RNF)

RNF1: Modularidade e Expansibilidade

O sistema deve ser construído com interfaces para ser facilmente expandido.

  • RNF1a: O sistema deve utilizar uma arquitetura baseada em módulos, onde cada módulo (ex: autenticação, relatórios, CRUDs) é independente.
  • RNF1b: Cada funcionalidade principal deve ser definida por uma interface, permitindo a substituição de implementações sem afetar o restante do sistema.

RNF2: Segurança de Credenciais

Deve haver mecanismos para verificar credenciais e alertar sobre acessos suspeitos.

  • RNF2a: As senhas de usuários devem ser armazenadas de forma segura, usando técnicas como hashing com salt.
  • RNF2b: O sistema deve monitorar e registrar tentativas de login falhas, alertando o administrador sobre padrões suspeitos (ex: múltiplas tentativas em um curto período de tempo de diferentes IPs).

RNF3: Proteção de Dados Privados

Chaves e dados privados devem ser inacessíveis e protegidos.

  • RNF3a: O sistema deve garantir que dados sensíveis (chaves de API, segredos) não sejam expostos em arquivos de configuração ou código-fonte.
  • RNF3b: Acesso a dados privados deve ser restrito, com um controle de acesso rigoroso que impede que usuários não autorizados visualizem essas informações.

RNF4: Confiabilidade e Feedback

O sistema deve fornecer feedback claro e confiável sobre o status do login.

  • RNF4a: O sistema deve exibir mensagens de erro específicas e amigáveis (ex: "Nome de usuário ou senha inválidos"), em vez de mensagens genéricas.
  • RNF4b: Durante o processo de login, o sistema deve fornecer um indicador visual (ex: um spinner de carregamento) para mostrar que a requisição está em andamento.

RNF5: Alta Disponibilidade

Para usuários internos, o sistema deve ser altamente disponível e enviar pings de aviso se houver problemas.

  • RNF5a: O sistema deve ter um mecanismo de heartbeat ou "ping" para monitorar a disponibilidade de seus serviços principais.
  • RNF5b: Se um serviço crítico falhar, o sistema deve enviar automaticamente uma notificação de alerta (ex: e-mail, mensagem) para a equipe de suporte.

RNF6: Limite de Dados na Versão Pública

A versão pública do sistema deve ter um limite imposto na quantidade de dados.

  • RNF6a: O sistema deve limitar o número de registros que podem ser criados ou visualizados na versão "Pública".
  • RNF6b: O sistema deve exibir uma mensagem clara ao usuário quando o limite de dados for atingido, orientando-o sobre as limitações da versão.

RNF7: Ausência da View Admin

A versão pública do sistema não deve conter a interface de administração (view_admin).

  • RNF7a: O sistema deve desabilitar completamente o carregamento da view_admin quando estiver no modo "Público".
  • RNF7b: Tentativas de acesso direto a URLs ou endpoints da view_admin no modo "Público" devem ser bloqueadas e retornar um erro de acesso negado.

RNF8: Responsividade

As interfaces de usuário devem se adaptar a diferentes tamanhos de tela (desktops, tablets e celulares) sem perda de funcionalidade ou usabilidade.

  • RNF8a: O layout da interface deve ser fluido e ajustável, utilizando media queries ou frameworks CSS modernos.
  • RNF8b: Todos os elementos interativos devem ser facilmente clicáveis ou tocáveis em telas menores.

RNF9: Desempenho Otimizado

As consultas e operações de CRUDs devem ser executadas em um tempo aceitável, mesmo com grandes volumes de dados.

  • RNF9a: O tempo de resposta para operações de leitura (leitura de dados, geração de relatórios) deve ser inferior a 2 segundos para a maioria dos casos de uso.
  • RNF9b: As operações de CRUD devem ser executadas de forma assíncrona para não bloquear a interface do usuário.

RNF10: Mecanismo de Backup e Recuperação

O sistema deve ter a capacidade de realizar backups automáticos dos dados e permitir a restauração completa.

  • RNF10a: O sistema deve ser configurado para realizar backups completos dos dados em intervalos regulares (ex: diariamente).
  • RNF10b: O processo de restauração deve ser simples e bem documentado, permitindo que a equipe de suporte recupere os dados rapidamente em caso de falha.

RNF11: Auditabilidade

A arquitetura do sistema deve facilitar a auditoria de segurança.

  • RNF11a: O sistema deve manter logs de todas as operações sensíveis (ex: alterações de permissões, exclusões de dados).
  • RNF11b: Os logs de auditoria devem ser imutáveis e protegidos contra modificações.

RNF12: Documentação Técnica Completa

Deve haver documentação clara e atualizada sobre a arquitetura do sistema, APIs, e como realizar a manutenção e o desenvolvimento futuro.

  • RNF12a: A documentação deve incluir diagramas de arquitetura, descrições de interfaces e módulos.
  • RNF12b: A documentação da API deve ser detalhada, incluindo exemplos de requisições e respostas.

RNF13: Interoperabilidade

O sistema deve ser capaz de se comunicar e trocar dados com outros sistemas por meio de APIs ou outros protocolos de comunicação padrão.

  • RNF13a: O sistema deve expor uma API RESTful para permitir a integração com sistemas externos.
  • RNF13b: A API deve ser documentada e segura, usando tokens de autenticação ou chaves de API.

RNF14: Acessibilidade

A interface de usuário deve ser projetada de acordo com as diretrizes de acessibilidade para a web (como as WCAG).

  • RNF14a: A interface deve ser navegável usando apenas o teclado.
  • RNF14b: Todos os elementos visuais devem ter um contraste de cores adequado e as imagens devem ter descrições (alt text) para leitores de tela.

RNF15: Localizabilidade

Deve ser possível adaptar a interface de usuário, mensagens e formatos de data/hora para diferentes idiomas e regiões.

  • RNF15a: O código deve separar o texto da interface em arquivos de recursos (.json, .properties, etc.) para facilitar a tradução.
  • RNF15b: O sistema deve formatar datas, números e moedas de acordo com as convenções da região do usuário.

RNF16: Manutenibilidade

O código-fonte deve ser limpo, bem comentado e seguir padrões de design estabelecidos.

  • RNF16a: O código deve ter uma estrutura lógica e consistente, facilitando a leitura e compreensão por novos desenvolvedores.
  • RNF16b: O sistema deve ter um conjunto de testes unitários e de integração para garantir que novas alterações não introduzam bugs.

RNF17: Escalabilidade

A arquitetura do sistema deve ser projetada para lidar com um aumento futuro no número de usuários, volume de dados e requisições.

  • RNF17a: O sistema deve ser capaz de ser implantado em múltiplos servidores para distribuir a carga (escalabilidade horizontal).
  • RNF17b: O banco de dados deve ser otimizado (ex: índices) para garantir que o desempenho não degrade com o aumento do volume de dados.
